## 🚀 执行流程

### 步骤 1: 代码规范审查

#### 项目规范
- [ ] 符合 CLAUDE.md 规范、PSR 标准、代码风格一致
- [ ] 命名规范正确、文件位置和命名空间正确

#### 代码风格
- [ ] 命名清晰、注释适当、函数长度合理（< 50 行）
- [ ] 类职责单一（SRP）、代码复用（DRY）

### 步骤 2: 安全性审查

#### 输入验证
- [ ] 所有用户输入都经过 Validate 验证器验证
- [ ] LIKE 查询中的 `%` 和 `_` 已转义

#### 敏感数据保护
- [ ] **Model 已定义 `$hidden`**，至少隐藏 `password`、`last_login_ip`
- [ ] **`$fillable` 不含特权字段**（`status`、`last_login_at`、`last_login_ip`）
- [ ] API 响应中不包含密码哈希、Token、密钥

#### 认证与授权
- [ ] **路由组已挂载认证中间件**
- [ ] `user_id` 从 JWT Token 获取，非 `request->param('user_id')`
- [ ] 资源所有权验证

#### 事务安全 ⚠️⚠️⚠️
- [ ] **≥2 个写操作已用 `Db::startTrans()` 包裹**
- [ ] **无 TOCTOU 竞态条件**（check-then-act 在同一事务内）
- [ ] 事务异常时正确 `Db::rollback()`

#### 其他安全
- [ ] 无 XSS/CSRF 风险、无硬编码密钥、错误信息不泄露敏感信息

### 步骤 3: 性能审查

- [ ] 无 N+1 查询、使用 Eager Loading
- [ ] 索引合理、查询优化（select 字段, limit）
- [ ] 大数据集使用分页、无内存泄漏

### 步骤 4: 可维护性审查

- [ ] 变量/函数命名自解释、逻辑流畅
- [ ] 依赖注入正确、无硬编码依赖、可 Mock 外部服务
- [ ] PHPDoc 注释完整、复杂业务逻辑有注释

### 步骤 5: 最佳实践审查

- [ ] Service/Repository Pattern 使用恰当、不过度设计
- [ ] 异常处理完整、日志记录合理、事务使用正确

### 步骤 6: 标记阶段完成 🚨

**必须执行 mark-stage 命令！** 见上方 CRITICAL 部分。

**严重问题定义**（发现任何一项必须标记失败）：
- 致命安全漏洞（SQL 注入、认证缺失、敏感数据泄露）
- 致命正确性问题（跨层调用参数不匹配、调用了不存在的方法）
- 事务安全缺陷（多步写操作无事务、TOCTOU 竞态）
- 模型安全缺陷（缺少 `$hidden`、`$fillable` 含特权字段）
- 严重性能问题、违反核心规范
